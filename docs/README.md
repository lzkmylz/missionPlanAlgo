## 一、调度流程概述

Greedy调度器的调度流程分为以下主要阶段：

```
1. 初始化阶段
   ├── 加载场景数据（卫星、目标、地面站）
   ├── 加载可见性窗口缓存
   ├── 初始化约束检查器
   └── 预计算卫星位置（可选）

2. 任务准备阶段
   ├── 创建观测任务列表
   ├── 按优先级/截止时间排序
   └── 初始化资源跟踪

3. 主调度循环（对每个任务）
   ├── 3.1 候选收集（阶段1）
   │   ├── 获取可见窗口
   │   ├── 能力检查
   │   ├── 时间窗口约束检查
   │   └── 资源约束预检查
   ├── 3.2 批量姿态预计算（阶段2）
   ├── 3.3 姿态预筛选（阶段3）
   └── 3.4 精确约束检查（阶段4）
       ├── 姿态复位机动计算
       ├── 任务机动计算
       ├── SAA检查
       ├── 资源约束检查
       └── 时间冲突检查

4. 结果整理阶段
   ├── 计算makespan
   ├── 频次满足度分析
   └── 生成报告
```

## 三、性能瓶颈分析

### 3.1 瓶颈1：精确姿态机动计算（占比~90%）

**问题描述**：
- 每个候选窗口需要进行2次精确机动计算
- 每次计算涉及刚体动力学、轨迹规划、能量模型
- 单次计算耗时5-10ms

**代码位置**：
```python
# scheduler/constraints/precise_slew_constraint_checker.py:322-330
reset_maneuver = calc.calculate_slew_maneuver(
    prev_attitude=current_state,
    target_attitude=nadir_attitude,
    current_time=prev_end_time
)
```

### 3.2 瓶颈2：候选数量过多（30万+）

**问题描述**：
- 1000目标 × 60卫星 × 5窗口 = 30万候选
- 即使大部分被预筛选过滤，仍需检查数千个
- 每个检查都需要完整的约束计算

### 3.3 瓶颈3：Python循环开销

**问题描述**：
- 当前实现是纯Python循环
- 每次迭代有函数调用开销
- 30万次 × 2次机动计算 = 60万次函数调用
